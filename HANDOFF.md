# FV「スワイプで空間を移動する」Three.js 実装 — 引継ぎ仕様書

## Claude Code への最初の指示(コピペ用)

> このリポジトリの `HANDOFF.md` を読んでください。`reference/fv-swipe-demo.html` が体験仕様の正となるリファレンス実装(CSS transform版)です。これと同じインタラクション仕様を維持したまま、Three.js + カスタムシェーダーで「深度マップによる2.5Dパララックス」と「シェーダー製シーン遷移」を加えたFVを実装してください。まず HANDOFF.md の受け入れ基準を読み、実装計画を提示してから着手してください。

## 1. 背景とゴール

建築・店舗系サイトのファーストビュー(FV)として、内観パース画像の中を「スワイプで歩いて見回す」体験を作る。参考にしたのは、PCに表示した内観パースをスワイプ操作で動かすTikTok動画の演出。

重要な前提: **自動では一切動かない。** 視点の移動はすべてユーザーのスワイプ / ドラッグ操作によって起こる。

現状はCSS transformによる平面パン+クロスフェードで実装済み(`reference/fv-swipe-demo.html`)。これをThree.js化し、以下を追加するのが本タスク。

1. **深度マップ2.5Dパララックス** — スワイプ時に手前の家具と奥の壁が異なる量で動き、実際にカメラが動いているような立体感を出す
2. **シェーダー製シーン遷移** — シーン1(ラウンジ)→シーン2(ライブラリ)の切り替えを、深度を利用したディスプレイスメント遷移にする

## 3. 体験仕様(リファレンス実装が正)

`reference/fv-swipe-demo.html` をブラウザで開き、必ず実際に触ってから着手すること。維持すべき挙動:

### パン操作
- ドラッグ / スワイプの移動量 × 約1.1 の係数で視点が水平移動(垂直は×0.7で控えめに)
- 表示はlerp(係数 ≈ 0.085)で追従し、リリース時は速度×12px程度の慣性で滑って止まる
- 画像は表示領域より約12%大きく(オーバースキャン)、端で見切れない
- 水平パン量に応じて ±1.8° 程度のわずかな rotateY(パース感の演出)

### シーン遷移
- 画像の端に達した状態でさらに同方向へスワイプすると「引っ張り」状態になる
- 引っ張り量に応じて次シーンがプレビュー表示される(現状: 不透明度 最大0.55)
- 引っ張りが閾値 **90px** を超えてリリース → 遷移発火。閾値未満ならスプリングバック
- 遷移時間 0.85秒、easeInOutCubic。入ってくるシーンは反対側の端寄り(maxX × 0.62)から始まり、同方向へスワイプを続けられる
- 遷移中の入力は無視。シーンは現在2枚だが、**N枚に拡張可能な設計にすること**

### スクロールとの共存(モバイルで特に重要)
- `touch-action: pan-y` + 移動方向判定(8px超えた時点で水平/垂直をロック)
- 水平ドラッグと判定された場合のみ `preventDefault` + pointer capture
- 縦スワイプは通常のページスクロールとしてそのまま通す

### UI(現状のDOMを流用してよい)
- ロゴ / ナビ / キャッチコピー(ダミー「KOMOREBI」。差し替え前提)
- 右下: 操作ヒントテキスト、視点位置ゲージ(現在のパン位置を示すライン)、シーン番号 `01 / 02`
- 左下: SCROLL インジケーター
- 初回ドラッグでヒントが `SWIPE TO MOVE ON` に変化

### アクセシビリティ・その他
- `prefers-reduced-motion: reduce` 時: lerp・慣性・遷移アニメーションを無効化(即時切り替え)。パララックス強度も0に
- キーボード ← → でパン、端でさらに押すとシーン遷移
- 画像に適切な `alt` 相当の代替情報(canvas利用のため `aria-label` 等で空間の説明を付与)

## 4. Three.js 実装方針(推奨。合理的な代案があれば計画段階で提案可)

### 構成
- フルスクリーンの `PlaneGeometry`(or fullscreen triangle)+ `ShaderMaterial` 1枚で描画。シーンごとにメッシュを分けるより、uniformで2シーンをミックスする方がシンプル
- uniforms(目安): `uTexA, uDepthA, uTexB, uDepthB, uPan(vec2), uParallax(vec2), uProgress(float), uDirection(float), uResolution, uImageRes`
- カメラは `OrthographicCamera` 固定。パンはuv空間で行う

### 深度パララックスのシェーダー(コア部分の考え方)
```glsl
float depth = texture2D(uDepth, uv).r;          // 0=奥, 1=手前
vec2 offset = uParallax * (depth - 0.5);        // 手前と奥で逆方向にずれる
vec3 color = texture2D(uTex, uv + offset).rgb;
```
- `uParallax` はパン速度・ドラッグ量から算出(最大 0.04〜0.06 程度。やりすぎるとエッジが破綻する)
- エッジの引き伸ばし対策として uv を中央寄りに 2〜3% スケールしておく
- 余裕があれば2〜3回のレイマーチ的サンプリングでオクルージョン感を上げてよい(必須ではない)

### cover表示とパン
- 画像ARと画面ARから cover 相当のuvスケールを計算し、`uPan` でuvを平行移動。可動域はCSS版の maxX / maxY に相当する量をuv空間で再現
- パン位置・慣性・引っ張り量などの**状態管理ロジックはリファレンス実装のJSをほぼ移植できる**(transform適用部分をuniform更新に置き換えるイメージ)

### シーン遷移シェーダー
- `uProgress`(0→1)で A→B をミックス。単純な `mix()` ではなく、深度を使ったディスプレイスメントを入れる:
  - 例: 出ていくシーンは `uv + uDirection * depthA * progress * strength` でめくれるように歪み、入ってくるシーンは逆方向から戻ってくる
  - 「奥へ歩いて進む」感を出すため、進行に応じてわずかなスケール(ズーム)も併用
- リリース前の「引っ張りプレビュー」も同じuniformで `uProgress` を 0〜0.3 程度まで部分適用すれば実現できる

### パフォーマンス・フォールバック
- `renderer.setPixelRatio(Math.min(devicePixelRatio, 2))`
- 描画は入力・アニメーション中のみ(完全に静止したらrAF内で描画スキップ可)
- テクスチャは `LinearFilter`、`generateMipmaps: false` で十分
- WebGL非対応 / コンテキスト生成失敗時は `reference/fv-swipe-demo.html` のCSS版にフォールバック
- 目標: ミドルレンジのスマホ実機で60fps、FVのJS+テクスチャ込み初期転送を常識的な範囲に(画像は遅延読み込み・圧縮を検討)

### 技術スタック
- Vite + npm の `three`(ESM)。バニラJSでよい(フレームワーク不要)
- 後で実サイト(未定: 静的 / WordPress等)に移植しやすいよう、FV部分を自己完結したモジュール(`initFV(container, scenes[])` のような形)にする

## 5. 受け入れ基準

- [ ] 自動の動きが一切ない(放置しても完全に静止)
- [ ] スワイプで深度パララックスが効き、手前と奥の動きの差が視認できる
- [ ] 端まで引っ張る→閾値超えリリースでシーン遷移、未満ならスプリングバック
- [ ] 遷移中に破綻(白フラッシュ、エッジの大きな乱れ)がない
- [ ] iOS Safari / Android Chrome 実機で縦スクロールが阻害されない
- [ ] `prefers-reduced-motion` で動きが止まる
- [ ] キーボード操作可
- [ ] シーンを3枚以上に増やしてもコード変更が配列追加だけで済む
- [ ] WebGL不可環境でCSS版フォールバックが表示される
